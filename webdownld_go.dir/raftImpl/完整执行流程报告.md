# webdownld_go.dir/raftImpl - 完整执行流程报告

## 命名说明（与论文对照）

- **`leader_last_idx`**：Leader 下一次要向该 Follower **从哪条日志下标开始发送**（论文常称 `nextIndex`）。
- **`follower_matched_idx`**：该 Follower 上**已确认复制到的最后日志下标**（论文常称 `matchIndex`）。
- **`follower_last_idx` / `follower_last_term`**（RPC 内）：追加新条目前，Follower 必须在「该下标」处与 Leader **前缀任期一致**（gRPC proto 字段仍为 `PrevLogIndex` / `PrevLogTerm`，与论文 `prevLogIndex` / `prevLogTerm` 对应）。

## 0. 启动

入口：`cmd/raftnode/main.go` 的 `main()`。

1. 解析 `-id`、`-raft`、`-http`、`-persist`、`-bootstrap` / `-join`、`-db`。
2. 打开 Pebble，启动协程消费 `apply_ch`（main 中局部变量名可为 `applyCh`），将已提交 `put`/`del` 应用到 Pebble。
3. 调用 `raft.NewRaft(id, raftAddr, persistPath, applyCh)`：
   - 读持久化（若有）；
   - `start_raft_grpc` 监听 gRPC，注册 `InternalRaft`；
   - 启动 `apply_committed_entries_loop`、`ticker_loop`。
4. 注册 Gin：`/status`、`/kv/*`、`/cluster/join`；非 bootstrap 时 `registerSelf` 向 Leader HTTP 发 join。
5. 等待信号后 `Shutdown` HTTP、`rf.Stop()`。

## 1. 用户写 KV（PUT）

1. `PUT /kv/:key` → `rf.Submit(Op{put})`。
2. Leader 追加日志、`try_commit`、`broadcast_append_to_followers`；Follower `AppendEntries` 复制。
3. `commit_index` 推进后 `apply_committed_entries_loop` 发 `ApplyMsg`，Pebble `Set`。
4. `Submit` 在对应索引提交后返回（内部 `submit_and_wait_applied`）。

## 2. 选举与心跳

- Follower 选举超时 → `start_election` → `RequestVote`。
- Leader 按 `heartbeat_due` 周期 `broadcast_append_to_followers`（空条目即心跳）。

## 3. 动态加入

- 新节点 `POST /cluster/join`（打到 Leader）→ `JoinNode` → `join` 日志复制 → 各节点 `apply_join_locked` 更新 `peers` 并建 gRPC 客户端。

## 4. 调用链（摘要）

```
main → NewRaft → start_raft_grpc → Serve
     → ticker_loop → tick → broadcast_append_to_followers / start_election
Submit → submit_and_wait_applied → apply_committed_entries_loop → apply_ch → db.Put/Delete
```

---

*与仓库源码一致；详细见 `internal/raft/*.go`。*

## 5. 网盘集成与一致性边界

1. Follower 的 HTTP 409 响应携带 `leader_id` 和内部 `leader_raft_addr`。
2. 网盘侧按 `leader_id` 映射 `RAFT_HTTP_NODES` 后重试，避免访问错误端口。
3. `Submit` 等待多数派提交以及 Pebble apply 完成。
4. 当前没有 ReadIndex、快照、联合一致成员变更，也未持久化 `commitIndex/lastApplied`。

## 6. 测试布局
